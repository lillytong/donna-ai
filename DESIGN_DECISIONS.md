### DD-01: Database is the source of truth; content stored as structured semantic data
**Decision:** The DB holds contract content as structured semantic data — prose as lightweight markup, tables as rows/cells — not as plain text and not as Word/OOXML fragments. Word is an export format.
**Rejected:** (a) plain-text bodies — loses tables, inline emphasis, and structure; (b) storing OOXML fragments — mixes presentation with data, hard to edit/query/diff.
**Rationale:** A *content-complete* data layer preserves meaning at zero drift while keeping the layer pure, queryable, diffable, and AI-native. Nothing of meaning lives only in formatting.

### DD-02: Presentation and numbering are derived, not stored
**Decision:** Formatting (bold/underline/highlight) is produced by deterministic rules from a **per-contract style config**. Clause numbering is **derived from tree position** + the numbering scheme.
**Rejected:** Storing numbers and formatting as content.
**Rationale:** Representative contracts are auto-numbered already; deriving numbers means add/move/delete propagates automatically and cross-references follow (DD-11). Per-contract style configs are required because house styles differ across documents.

### DD-03: Tracked changes come from DB snapshot diffs
**Decision:** On export, diff the previous snapshot against current node state; emit as Word tracked-change markup. Pure renumber shifts are **suppressed** (not shown as changes).
**Rationale:** We generate .docx from the DB, so Word's native tracking is unavailable. DB diffs are better — we control granularity and avoid polluting redlines with dozens of spurious "2.3 → 2.4" marks.

### DD-04: Import is two-step, with two modes
**Decision:** (1) AI-assisted parse → candidate structured tree; (2) human review/correction before commit. **v1 first import is clean-only** (fresh draft); a clean-document guard blocks/flags tracked changes on import (DD-46). The resolve-tracked-changes first import (heavily redlined doc, two-tier triage) is **deferred to v2** (DD-34, DD-46) — in the normal lifecycle donna.ai is present from v1 and every counterparty redline flows through Mode B (DD-25/DD-28).
**Rejected:** Single-step auto-import; building Mode A tracked-change triage in v1.
**Rationale:** Legal parsing is structurally ambiguous; a wrong parse silently corrupts everything downstream. Tracked-change volume (zero to 700+) is handled in Mode B; first import is clean in v1, so the import spine stays small (DD-46).

### DD-05: One adjacency-list node tree (clauses + appendices unified)
**Decision:** A single self-referential `nodes` table of arbitrary depth. Appendices are top-level branches that nest like Articles. Each node has a content_type (prose/table/attachment).
**Rejected:** A separate appendices entity; a JSONB tree blob.
**Rationale:** Appendices have their own sections/sub-sections and must be selectable, issue-able, and editable exactly like clauses. Per-node rows enable per-node querying, embeddings, issues, and versioning.

### DD-06: Tiered context injection — see §7. Structured data = DB query always. Free-text prose = semantic search. `node_embeddings` (intra-contract semantic search) built in Phase 2 regardless of volume — required for Donna's analytical intelligence, not a scale feature. `comment_embeddings` (negotiation history search) deferred to Phase 2+ when comment prose exceeds ~100K tokens. See DD-32.

### DD-08: Call mode merged into edit mode
**Decision:** No separate call UI. Live calls use the same select-node → issue/edit flow.
**Rationale:** The standard flow already covers live capture; a separate mode adds complexity for no new capability.

### DD-09: Snapshots as the versioning primitive
**Decision:** `contract_snapshots` capture all node states at send-time, like git commits.
**Rationale:** Required for accurate redline diffs; also reproduces exactly what the counterparty reviewed on any date, and powers version-diff views.

**Storage-model addendum (full-copy per snapshot — Lilly-affirmed 2026-06-25).** Each snapshot stores a **complete self-contained JSONB dump of the whole tree** (`contract_snapshots.tree` — topology + every node's body), NOT a delta against the prior snapshot and NOT content-addressed/deduped. Unchanged clauses are therefore duplicated across snapshots — accepted deliberately. **Why full-copy:** (1) the snapshot is the legal record of "what we sent" (§2.4) — a self-contained blob renders back with zero reconstruction risk, where a delta-chain must replay deltas and one bad link corrupts the record; (2) it is **robust to structural edits** (delete / move / reorder / shuffle clauses) — a full dump just *is* the tree as it stood, whereas a delta scheme must encode structural moves correctly to reconstruct, the fragile part; (3) the granular per-edit deltas already live in `node_versions` (the redline/diff path), so the dump is purely the immutable baseline, not the diff source — we get both. **Scale rationale (Lilly's call):** single-operator, low contract volume (not a 1000-contract firm); snapshots are cut only on boundary events (Mark-as-sent / import) — ~5–20 per contract over its life; a contract is a few hundred KB, TOAST-compressed → single-digit MB per contract worst case. The duplication cost is negligible at this scale; immutability + read-simplicity + structural-edit robustness win. **Rejected (deferred, not wrong):** content-addressed node storage (hash bodies, snapshot = manifest of node-hashes → dedup unchanged nodes, Git's object model) — the clean dedup upgrade, but adds a reassembly layer + GC story. **Revisit trigger (the ONLY one):** materially higher volume — many contracts × frequent snapshots making the bytes actually matter. Until that changes, full-copy stands; do not "optimize" it into deltas/dedup absent that trigger. **Known fidelity gap (separate from storage, tracked):** the dump currently omits `table_data`, so table clauses aren't fully captured in a snapshot — to close when the redline path is hardened.

**Decision:** Extract defined terms on import into a deal-scoped registry; link mentions in node bodies.
**Rationale:** Prevents silent cross-contract divergence, enables hover-to-define and accurate AI context.

### DD-11: Cross-references as structured links

### DD-16: Import review UI is a tree editor with multi-select
**Decision:** The import review UI renders the parsed candidate tree as an interactive tree editor. Operator can select multiple nodes and shift their depth level in one action. Nodes the parser flagged as low-confidence are highlighted. No commit until the operator approves.
**Rationale:** Lawyer-drafted contracts arrive with inconsistent formatting — manually typed numbers, mixed styles, non-standard heading levels. The error rate on first parse is real but bounded (~5-10%). Multi-select bulk correction makes fixing 20-30 nodes fast enough that it's not a barrier to import.

### DD-17: Two issue anchor modes — node-scoped and contract-level free-floating
**Decision:** Issues can anchor to a specific node (standard) or float at contract level with no node anchor (`node_id = NULL`). Free-floating issues capture general remarks, structural concerns, and negotiation points raised before the relevant clause exists.
**Rejected:** Forcing every issue to anchor to a node.
**Rationale:** Live negotiations surface points that don't map to existing clauses — "we need a whole new section on X," "generally uncomfortable with the liability framework." These must be captured immediately without blocking on clause creation. Free-floating issues can be anchored to a node later, or remain contract-level if they resolve without a clause change. **Anchor at creation defaults to the clicked node and is editable; the anchor can be re-pointed at any time (`node_id` mutable).** For a not-yet-existing clause, anchor the issue to the nearest existing parent (or leave it document-level), then re-point it to the new node once the clause is inserted.

### DD-18: Node creation mid-negotiation
**Decision:** Operator can create a new node at any point, choose its parent, and position it among siblings. The node immediately receives a derived number, is editable, and can have issues attached. It surfaces as a tracked insertion in the next export.
**Rationale:** Negotiations routinely add new sections. The tool must support the live workflow without forcing a re-import or a Word round-trip to add a clause.

### DD-19: Left panel is modal, not contextual.

The context panel (slide-in on clause selection) was considered and rejected. It hijacks screen real estate during live calls without explicit intent. The left panel toggle is explicit — user chooses when to switch. Right panel is never displaced by a context panel.

### DD-20: Issue visibility is pull, not push.

Issues are not surfaced inline in the document view by default. The operator signals intent (toggle to issues mode, open the issues tab) to see them. During a live call, the document view is sacred.

### DD-21: Style config is per-contract, set at import, auto-applied on every export.

Rejected alternative: style options at export time. Reason: export must be one click; formatting decisions made repeatedly are formatting decisions made inconsistently. Style templates (shared configs) reduce setup work across contracts in the same deal.

### DD-22: Every export cuts a snapshot; "sent to counterparty" is a separate explicit tag.

Snapshots are automatic on every export — no action required, version history is always complete. The "sent to counterparty" tag is manual: operator checks a box on the export that is actually going to the counterparty. The tag is exclusive — only one snapshot holds it; applying it moves it from the previous holder. Mode B always diffs against the tagged snapshot. Tag can be moved retroactively via snapshot history. Internal exports (sharing with a lawyer, internal review) cut a snapshot but do not affect the diff baseline. **Rejected alternative:** tag every export as the counterparty baseline automatically. **Reason:** the operator regularly exports for internal use; conflating "exported" with "sent to counterparty" would corrupt the diff baseline on the next revision import. **Generalised by DD-48:** the single "sent to counterparty" tag becomes one of four named version pointers (shared / received × counterparty / legal); the export recipient selector sets the appropriate "shared with" pointer, and importing a revision sets the matching "received from" pointer. **Refined by DD-61:** the "internal exports cut a snapshot" clause above is superseded — only a **send** (Counterparty / Legal recipient) cuts a snapshot; Internal / Copy-only grabs cut none.

### DD-23: Contract types and clients are user-configurable in Settings, never hardcoded.

Rejected alternative: enum of contract types in application code. Reason: every operator's deal taxonomy is different; hardcoding makes the tool fragile for any client that doesn't fit the assumed types.

### DD-24: Style templates are reusable configs; contracts inherit and override.

Rejected alternative: copy-paste style config per contract manually. Reason: when a deal has three contracts sharing the same house style, the style should be defined once and inherited, not duplicated three times and then drift.

### DD-25: Counterparty revision uses a dedicated staging structure; issues are created only as outcomes, not upfront.

`counterparty_revision_changes` (one per node/chunk) and `counterparty_revision_hunks` (one per individual edit within a node) are the staging tables. **Navigation unit = node (chunk).** Decision unit = hunk. Chunk = node by definition: numbered clauses, lettered sub-clauses, free-floating paragraphs, definitions, and tables are all nodes and therefore all chunks — this is not a new concept, it is already embedded in the tree model. After the operator decides each hunk: accepted/modified → applied directly to the node, no issue created; rejected/deferred → issue created with `initiator: counterparty`, `their_position` = proposed text. Issues represent persistent contested points; the review tables are a staging workflow. Rejected alternative: surface every edit as an issue upfront. Reason: (a) accepted changes shouldn't become issues — they're resolved immediately; (b) grouping by node is the right cognitive unit — you cannot evaluate a word deletion without seeing the surrounding additions; (c) 40 individual issues for 12 edited nodes is overwhelming; 12 node-level items is manageable. Donna pre-classifies hunk significance by semantic impact, not edit size — a single word change can be substantive if it alters meaning, obligation, or commercial effect. Trivial = demonstrably no meaning change. Substantive = anything that shifts what a party must do, when, or under what conditions. Two rules: (1) classification requires DD-31 context resolution first — never classify on raw hunk text alone; (2) trivial is a high-confidence positive call, not a residual — when uncertain, always classify substantive. Trivial hunks are pre-recommended Accept with no counter drafted; substantive hunks get full analysis + exact counter-language regardless of how many words changed.

### DD-26: Counterparty revision review uses inline tracked-change rendering, not a two-pane diff.

Rejected alternative: left-right split (what you sent vs. what they changed). Reason: two-pane halves screen real estate, forces left-right eye movement, and is less familiar to contract readers than inline tracked changes. Strikethrough/underline inline is how Word shows it; operators already know how to read it.

### DD-27: Donna drafts exact counter-language on every counterparty-proposed change, not instructions.

Rejected alternative: high-level guidance ("you should reject this because X"). Reason: the value proposition is removing drafting work. The operator's job is to judge Donna's language, not write from scratch. Four actions: Accept theirs / Use Donna's counter / Edit Donna's counter / Keep original. **Positional grounding — no separate "brief" needed:** Donna infers the operator's position from what the system already knows: (1) agreed issues — what was ultimately accepted on settled points; (2) open issues — `our_position` on each contested point; (3) deal parameters — the locked commercial figures; (4) the draft contract itself — the starting language reflects the operator's opening position. On the first issue for a topic with no prior positions, Donna grounds in the draft language + live research (DD-38) for market data questions, and flags her reasoning basis explicitly rather than asserting a position she hasn't been given.

### DD-28: Counterparty revision diff uses two paths.

Path A: if incoming .docx contains Word tracked changes (`<w:ins>`/`<w:del>`), extract directly from XML — reliable, exact. Path B: if clean edited copy (no tracked changes), parse → match nodes against last snapshot (by section number, then heading similarity, then body text similarity) → diff text pair-by-pair. Both paths produce identical output: a numbered list of changes for the review flow.

### DD-29: Every accept/reject/modify decision is a learning signal, logged with rich context from day one.

Phase 2: Donna retrieves past decisions as RAG context — "In similar situations, you rejected this type of change." v2: Donna infers negotiation principles from accumulated decisions and applies them proactively without citing examples. The infrastructure (rich `decision` JSONB on issues) must be built in Phase 2 even though the intelligence matures in v2. If we don't log decisions richly from the start, v2 has nothing to learn from.

### DD-30: Both import modes are fault-tolerant for structural errors.

First import: tree editor review flags uncertain nodes in amber; operator corrects before commit. Counterparty revision import: a structural triage step runs before the content review flow, surfacing any formatting/depth/heading errors the counterparty's legal team introduced. Operator resolves structural issues first, then content decisions. The two concerns (structure vs. content) are deliberately separated — conflating them in one review flow creates confusion about what is being decided.

### DD-31: Donna resolves cross-references, defined terms, and deal parameters before analysing any clause.

When a node contains `{Term}`, `[[Section X.Y]]`, or `[figure]` markers, Donna retrieves the referenced content from the DB and injects it as context before classifying significance, drafting counter-language, or answering a question. This applies to all three Donna surfaces (counterparty revision reviewer, issue-scoped assistant, contract-scoped Q&A assistant). Depth limit: one level of reference resolution by default — Donna follows direct references from the clause under analysis but does not recursively chase reference chains. If deeper resolution is needed for a reliable answer, Donna flags it rather than hallucinating context. Cross-contract references (`[[TLA Section 4.2]]`) resolve the same way if the target contract is in the DB. Rejected alternative: analyse only the clause text in isolation. Reason: a percentage change in one clause is meaningless without knowing what the base it applies to means in another clause — shallow analysis produces shallow recommendations.

### DD-32: `node_embeddings` are built in Phase 2, not Phase 2+.

Rejected assumption: intra-contract semantic search is only needed at volume. Correct framing: Donna needs to discover implicit clause relationships within a single contract to reason reliably — explicit cross-references only cover links the drafter thought to mark. A payment clause may be semantically related to a price-adjustment clause with no `[[...]]` link between them. Donna must be able to find that relationship herself. pgvector on `node_embeddings` is therefore core to Donna's analytical intelligence, not a scale feature. The ~100K token threshold only applies to `comment_embeddings` (negotiation history search).

### DD-33: Embedding generation is always asynchronous, always diff-scoped, and always post-structure-finalisation.

Three rules that never bend: (1) never synchronous — embedding API latency must never block a save; (2) never re-embed unchanged content — only nodes whose `body` actually changed get queued; (3) never embed before structure is confirmed — embeddings fire only after the operator has committed the corrected tree (Mode A: after import review Commit; Mode B: after structural triage is committed, then per content decision). Embedding malstructured pre-correction content wastes API calls and pollutes search. Structural changes (renumber, reorder, reparent), defined term updates, and deal parameter changes do NOT trigger re-embedding. Staleness tracked via `embedded_at` vs `updated_at`; worker skips any node where `embedded_at >= updated_at`.

### DD-34: Tracked-change review in Mode A uses two-tier triage, not one-by-one review.

**[Deferred to v2 — superseded for v1 by DD-46; first import is clean-only. Retained for the v2 "onboard an existing in-flight contract" feature.]** A representative contract may contain 700+ tracked changes. Reviewing each individually is a non-starter. Two tiers: (1) **Bulk triage** — Donna pre-classifies all tracked changes as `accepted_baseline` (change is already agreed or is a trivial formatting difference — fold into the starting node text) vs `open_proposal` (live negotiation point — seed as issue). Operator sees the **full list of Donna's auto-accept candidates** — node reference, change type, original → proposed text inline — not just a count. Operator scans the list and **unchecks any item they want to pull out for individual review**. One "Commit triage" button: checked items fold into baseline, unchecked items join the open issues queue. (2) **Individual review** — the unchecked items (typically 20–50 of 700) are reviewed one by one in Tier 2, using the same four-action flow as Mode B counterparty revision review. Operator maintains full oversight and veto at all times; Donna proposes, operator approves. Rejected alternative: single-item sequential review of all changes. Reason: at 700 items, sequential review is not meaningfully different from doing it in Word — it eliminates the tool's value on the most painful import case.

### DD-35: AI model tier assignment by task consequence — see §7 model quality principle.

Product decision: Opus for high-consequence tasks (significance classification, counter-language, brainstorm, triage); Sonnet for medium-consequence tasks (Q&A, summaries, context resolution); Haiku for low-consequence extraction and detection. Haiku pre-screen router cuts Opus calls on bulk triage. ### DD-36: Numbering-pattern inference corrects hierarchy silently before structural triage.

After parse, before any operator-facing review, a post-parse pass reads each node's heading text and infers the correct parent-child relationship from its numeric or letter prefix. Inference rules: `N.` = depth 0 (article); `N.M` = depth 1, child of article N; `N.M.P` = depth 2, child of section N.M; `(a)`, `(b)` etc. = child of the immediately preceding decimal clause at depth N-1, resolved by position not by explicit parent reference. Where inferred depth contradicts parsed outline level, inferred depth wins — the heading number is ground truth. Nodes with no recognisable numbering prefix (body text, narrative paragraphs, table rows) retain their parsed depth; if their position looks anomalous relative to numbered siblings, they are surfaced in structural triage. The correction is silent — no operator action required unless residual anomalies remain after the pass. **Rejected alternative:** rely solely on Word's outline level / list structure. **Reason:** counterparty documents routinely apply heading styles inconsistently; the numbering in the clause text is the only reliable, document-agnostic signal for hierarchy.

### DD-38: Donna invokes live web research when an issue involves a market data point.

### DD-39: Intended negotiation cadence — legal version first, verbal discussion second.

When the counterparty's legal team sends a revised version AND the counterparty contact has verbal feedback, the correct sequence is: (1) legal redline arrives and is imported via Mode B — this is the diff baseline; (2) verbal call happens on top of the reviewed version — verbal points are additive issues, never a parallel track. **Rejected:** three-way merge (operator produces one version from verbal feedback; legal team produces another from the same base; tool reconciles both) — deferred to v2. **Rationale:** parallel editing creates two competing versions of the same clauses with no clean import path; the process constraint eliminates the conflict entirely at the cost of requiring counterparty cooperation on sequencing. ### DD-40: Donna tab is a persistent conversation, context-managed by sliding window + rolling summary.

The contract-level Donna Q&A thread (`donna_conversations` + `donna_messages`) persists across sessions — the operator sees prior questions and answers when she returns days or weeks later. **Context blowup is avoided structurally:** the full thread is stored in the DB, but only the **last 10 turns + a rolling `running_summary`** are injected into any Donna call (per the project's LLM windowing standard — sliding window of recent turns plus a running summary; never the full history). Turns older than the window are collapsed into `running_summary`, which Donna updates incrementally. Older turns remain queryable on demand (Donna can pull a specific past exchange from the DB) but never bloat the prompt. **Rejected:** (a) stateless tab — operator re-asks the same questions every round, loses continuity; (b) inject full history every call — context grows unbounded and cost scales with thread length. **The engineering ADR:** summary-update trigger, exact window size, summarisation prompt. (The product owner confirmed persistent + windowed.)

### DD-41: Contract amendments are new contracts under the same deal, not edits to the signed original.

A signed contract is frozen (`status: signed`, read-only, no further node mutation). When the agreement must change post-signature, an amendment/addendum is imported as a **separate contract** in the same deal, cross-referencing the original's clauses (`[[Master Agreement Section 4.1]]`). The deal stays active as long as any contract under it is live. **Rejected:** mutating the signed original. **Rationale:** the signed document is a legal record of what both parties executed — it must never change; the deal's existing one-to-many contract structure already models this with no new entity. The negotiation workflow on an amendment is identical to any other contract. (The product owner confirmed.)

### DD-43: Import and export are two transforms over one intermediate representation; export is a deterministic renderer that doubles as the import verifier.

The **skeleton** (`node_tree` + `style_config` + markup vocabulary) is the single IR both directions pivot on. **Export** = a pure, deterministic function `render(skeleton) → .docx` (numbering from tree position, formatting from `style_config` per depth, cross-refs from target position, markup resolved to text) — **no AI in the export path.** **Import** = a layered parse, deterministic-first: (1) OOXML → block stream, (2) blocks → flat node list, (3) numbering-prefix → tree topology (DD-36), (4) style-table inference for un-numbered nodes, (5) AI-assisted semantic enrichment (terms / cross-refs / params / markup, DD-35), (6) human review of layers 4–5 flags only. **Every import is self-verified by re-rendering and diffing content against the source** — drift is surfaced in the tree editor before commit. **One `style_config` rule table** drives export rendering, import detection, and the preview pane. **One markup resolver** serves import-save, export-render, and Donna's context injection (DD-31). **Rejected:** (a) separately-written import and export formatting logic — drifts, double-maintained; (b) import as a pure inverse of export — impossible, formatting→structure is many-to-one and lossy; (c) any AI in the export path — non-determinism breaks round-trip verification and content integrity (Principle 1). **Rationale:** export's determinism is what makes import *safe* — re-render-and-diff turns "did I parse correctly?" from unanswerable into a mechanical check, and confines AI to the genuinely ambiguous residue while the deterministic core (esp. numbering, the bridge derived on export and used as the primary structural signal on import, DD-02/DD-36) carries the load. Ties to OQ-08: the skeleton the round-trip operates on must already carry full tree topology, so the snapshot/topology question is a Phase-0 schema concern, not Phase-3. ### DD-48: Each contract carries four named snapshot pointers plus the live working copy.

Four timestamped pointers, each referencing at most one immutable snapshot and each advancing independently as boundary events occur: `last_shared_with_counterparty`, `last_received_from_counterparty`, `last_shared_with_legal`, `last_received_from_legal`. The **working copy** is the live, mutable node tree — not a snapshot, not a pointer; it is simply the contract as it currently is. **"Shared with X"** is set on export to X (recipient chosen at export) and does double duty: the automatic diff baseline for X's next inbound revision (DD-47 source → baseline) and the record of what/when was sent. **"Received from X"** is set on import of a revision from X and points to an **immutable snapshot of X's version exactly as received** (baseline + their proposed changes applied, captured *before* the operator reviews or edits — option (a), product-owner-confirmed). It is **frozen — it never moves when the operator later edits**; this is the load-bearing property that makes `diff(working_copy, last_received_from_X)` answer "what have we changed since their version." The operator may adopt a received version as the working basis (equivalent to accept-all in the Mode B review, or a deliberate shortcut); the instant she edits, the working copy diverges and the diff becomes meaningful — but a received pointer is itself never the working copy. A single snapshot may hold several pointers (the version shared with legal is both the current export and `last_shared_with_legal`). Importing a revision therefore materialises an as-received snapshot **in addition to** running the Mode B review: the reviewed/edited result becomes the working copy; the as-received snapshot stays frozen as the pointer. **Generalises DD-22's single exclusive "sent to counterparty" tag** into this family; the export recipient selector (counterparty | legal | internal | copy-only) sets the matching "shared with" pointer (**internal and copy-only set none — both are pure downloads that cut no snapshot, DD-61**). Timestamps come from snapshot `created_at` (shares) and import time (receives). Enables a **version-lineage view** (the v1→v2→…→vN chain with pointer labels) — the at-a-glance antidote to version divergence. **Rejected:** (a) "working copy" as a fifth co-equal tag — conflates live mutable state with frozen snapshots; (b) "received from X" pointing to the post-review merged state — loses the faithful record of what X actually proposed. **Caution (for build):** "adopt their version as working copy" means accepting all their changes — fine for a trusted source (own legal's cleanups) but a major concession from the counterparty; it must be a deliberate, logged accept-all through the review flow, never a silent overwrite. And with two parties interleaved, `diff(working, last_received_from_counterparty)` shows everything since the counterparty's version — including legal's later changes and the operator's — not only the operator's edits. (Confirmed (a) immutable-as-received; the working copy diverges on edit; diff the two for changes-since.)

### DD-54: Import distinguishes front-matter from operative clauses; not every paragraph is a clause.

**Problem (found in real-data review):** the parser categorized *every* paragraph as a clause/body — so the title page, parties, recitals, and every table-of-contents line imported as fake clauses, numbered 1, 2, 3… before the real first clause ("1. Definitions…") even began. **Decision:** the import recognizes three regions by role. The **operative-clause tree begins at the first top-level numbered heading** (e.g. "1. NAME"; the parser may generalize to "Article 1"/"Section 1" — the engineering detection ADR). Everything *before* that boundary is **front-matter**, carried on a new `nodes.role` field: the first block is `title`, the remainder `preamble`. Front-matter is **preserved and shown but excluded from the clause tree and clause numbering** — which also fixes the spurious 1/2/3 numbering, since the operative tree now re-derives from the real clause 1. The **table of contents is detected and dropped** on import — this is conformance to the existing §10 rule ("TOC regenerated on export, never stored"), so TOC-as-clauses was a bug, not a new behavior. **Full role taxonomy (validated against JVA/OA/TLA):** front-matter = `title` | `date` | `parties` | `recital` | `agreement_statement`; body = `clause`; back-matter = `appendix` (DD-05) | `signature_block`; cross-cutting = `drafting_note`. Two roles carry rules beyond labeling: **`drafting_note`** — internal counsel/author commentary in brackets (JVA ~28, TLA ~14 in the real set) — is kept but **never crosses into a counterparty export** (§12 integrity; a leaked internal note is a credibility failure); **`signature_block`** (all three contracts) is captured, not a clause. **TOC** is dropped (§10). **Placeholders** — fill-in blanks (`[insert …]`, `___`; JVA ~28, OA 5, TLA 6) — are an inline `has_placeholder` flag → pre-signing "incomplete field" alert (F28), not a role. Front-matter + `signature_block` + `drafting_note` are excluded from the clause tree and numbering; the operative tree re-derives from the first `clause`. **F04 impact:** non-clause roles render as labeled regions (front-matter above, signature/appendix below), not numbered rows; drafting-notes flagged distinctly. **Engineering (DEV_TODO):** boundary detection, the `role` column + `has_placeholder`, per-role detection, TOC exclusion, export-exclusion of `drafting_note`, and the F04 rendering.

**Classification — deterministic-first, AI only for the residue (confirmed).** Roles are assigned by cheap deterministic rules first (keyword/structure: `WHEREAS`→recital, `BETWEEN`→parties, `IN WITNESS`→signature_block, bracketed `[…Note:…]`→drafting_note, tab+page-number→TOC, first `1.`→clause boundary) — free, instant, no hallucination. Only blocks the rules can't confidently place go to an AI pass at the **low-consequence tier (Haiku, DD-35)** — structured classification whose output the operator verifies in F04 (the existing ⚠ mechanism), so the model tier matches the stakes. Not a new subsystem — the same deterministic-then-AI-then-operator-confirm pattern as the existing import detection. **Guard (`drafting_note` is special):** it is the one role excluded from the counterparty export (§12), so a mislabel is asymmetrically costly in both directions (drop real content / leak an internal note). Therefore content is **never auto-excluded from export on the model's say-so alone** — when uncertain whether a block is a drafting note, surface it for operator confirmation; never silently drop. (When-in-doubt-surface, the same stance as the trivial/substantive rule.)

### DD-55: Brainstorm sessions are analyzed on close to extract compact negotiation patterns; raw conversation still discarded; patterns merge-first, consolidate, and prune on TTL.

## DD-57 — Schema evolves via forward-only SQL migrations; `schema.sql` stays canonical, no Alembic

**Problem:** `schema.sql` is loaded only by docker-entrypoint-initdb.d, which runs **only on a fresh (empty) data directory**. During development the schema evolved (e.g. `nodes.role` added) but a running DB with data never received the change — caught when `/commit` 500'd on the missing column. The only recovery was drop-and-rebuild, which destroys data. Acceptable while there is no real data; a one-way door the moment a real contract is committed and kept. **Decision:** add a forward-only migration mechanism. `db/migrations/NNNN_description.sql` files hold incremental deltas; an async runner (`python -m backend.migrate`, `backend/migrate.py` — infra alongside `db.py`) applies every file not yet recorded in a `schema_migrations(version, applied_at)` table, in filename order, each in its own transaction. `schema.sql` **remains the canonical full snapshot** for fresh installs and additionally seeds `schema_migrations` with a `0000_baseline` marker, so a freshly-built DB is already-current and the runner is a no-op there. The maintenance discipline (documented in `db/migrations/README.md`): every schema change folds the delta into `schema.sql` **and** adds a migration file **and** stamps that version into `schema.sql`'s seed block. Adoption on an existing unmanaged dev DB (no real data yet) = one rebuild from the current `schema.sql`, which stamps the baseline; thereafter the DB is managed. **Rejected:** (a) **Alembic** — not in the declared stack (raw SQL + asyncpg, no ORM, by deliberate deviation), and its autogenerate/versioning would become a *second source of schema truth* alongside `schema.sql`, the exact coupling the project avoids (DD-01, `schema.sql` canonical); (b) **migrations-only, drop `schema.sql`** — would lose the single readable full-model snapshot the project relies on; (c) **idempotent re-runnable migrations with no version table** — brittle for CHECK-constraint changes and hides drift. **Rationale:** a tiny hand-rolled runner over plain `.sql` files preserves `schema.sql` as the one schema truth, adds no dependency, and gives a no-data-loss evolution path; the baseline-stamp keeps fresh installs and live DBs convergent without double-applying.

## DD-61 — Only a send cuts a snapshot; Internal/Copy-only exports are pure downloads

> **Amended by DD-71 (2026-06-25):** donna.ai cannot actually send (no email/transport), so there is no in-app "send" — **all exports are now grabs** (pure downloads, no snapshot/pointer/lineage), and the snapshot-cut + `shared`-pointer-advance move to a separate **"Mark as sent"** action the operator triggers after sending manually. The two intents below ("send" vs "grab") collapse: the *boundary event* is Mark-as-sent, not any export. The snapshot service is reused unchanged; only the trigger relocates. Consequences 1–2 below now key off the first Mark-as-sent rather than the first send-export; 3–4 are unchanged.

**Context.** DD-09/DD-22/DD-48 established "snapshot on export." Building F15b (clean copy) surfaced a real consequence: the snapshot cut also stamps the pending `node_versions` edits group under the new snapshot (the group F15 reads as "this round's changes"). So a `copy_only` download — the operator re-grabbing the current .docx mid-draft just to read it — would cut a snapshot **and** close out the current round's change set. Repeated grabs fragment one round into many tiny snapshots, littering the version-lineage view (F27) with phantom versions and muddying what the next redline (F15) diffs as "this round."

**Decision.** The Clean-copy recipient selector has two intents, not four:
- **Send** — recipient **Counterparty** or **Legal**. Cuts a snapshot, stamps the pending `node_versions` edits group under it (closing this round's change set), and advances that party's `last_shared_with_X` `shared` pointer (DD-48). This snapshot is both a redline baseline and a lineage node.
- **Grab** — recipient **Internal** or **Copy-only**. Regenerates the current contract through the style config and downloads it. **No snapshot cut, no pointer moved, no edits-group stamped, zero lineage effect.** The operator re-reading their own working copy is not a new version.

**Consequences pinned for build (F15 depends on these):**
1. The redline baseline (`last shared with counterparty`, DD-48) advances **only on a send**, so it is unambiguous — intervening grabs never move it, and the round's change set stays whole as one group until the next send closes it.
2. Redline stays **disabled until a snapshot exists**; only a send can create the first one (a Copy-only/Internal grab never satisfies the gate).
3. Export sets `shared` pointers only. The `received` pointer is set **solely by the Phase-2 revision-import path**, pointing at an immutable as-received snapshot captured before edits (DD-48) — the export/send path never touches it.
4. The `internal` value in `snapshot_pointers.party` takes no pointer in v1 (Internal export is a pure download); left dormant for a future multi-user internal-review feature, not removed.

**Rejected:** (a) **Keep cutting a snapshot on every export, accept the fragmentation** — corrupts the lineage "where are we" briefing and the round change-set with phantom re-read versions (§2.4 trust: the lineage the operator briefs a principal from must show real rounds, not file-grabs). (b) **Cut a snapshot but skip only the pointer/edits-group stamping on grabs** (the prior §9 "cuts a snapshot, advances no pointer" wording) — still litters lineage with versions that are byte-identical to the working copy and carry no boundary meaning; the snapshot has no consumer, so cutting it is pure noise. (c) **Fold Internal into the Legal pointer** — Internal is not an external diffable party; in single-operator v1 (DD-53) there is no second internal reviewer in the system, so it has no baseline and no lineage value.

**Rationale.** A snapshot is a record of a boundary event — a version that went out to a party, or a revision that came in. A file re-grab is neither. Restricting the cut to sends keeps the redline baseline unambiguous, the round change-set intact, and the lineage view trustworthy (§2.4) — and collapses a confusing four-way "what does each recipient do to history" into a clean **send-vs-grab** primitive. **Supersedes** the "Internal exports cut a snapshot" clause in DD-22 and the §9 "Internal/Copy-only cuts a snapshot but advances no pointer" wording.

## DD-63 — Deleting a contract preserves deal-shared data (delete-vs-SET-NULL)

**Context.** Phase-0 `delete_contract` cascades a contract's own children (`issues → footnotes → node_versions → nodes → contract`; the `issue_comments` step is gone — the table was removed by DD-67). But two entity types are **deal-scoped, not contract-scoped**, and a delete would otherwise orphan or break sibling contracts in the same deal: `defined_terms` (one term set shared across the deal) and `cross_references` (can point from a clause in one contract to a clause in another contract in the same deal). Phase 2 (Mode B versioning) makes deletion live — it adds snapshots and a version lineage that must stay coherent — so the semantics must be pinned before that phase. No production data exists yet, so this is a forward decision, not a migration.

**Decision.** A contract delete is **scoped to the contract's own subtree**; deal-shared data is preserved:
1. **Cascade** only the deleted contract's own rows: its `nodes`, `node_versions`, `issues`, `footnotes`, and its own `contract_snapshots` / `snapshot_pointers`. (`issue_comments` was in this list before DD-67 removed the table; subsequent linked tables — `donna_recommendations`, `brainstorm_summaries` (DD-77) — extend the same cascade.)
2. **SET NULL** any `cross_references` (from either side) that pointed *into* a deleted node, so a sibling contract that referenced the deleted one keeps its clause text with a dangling-link marker rather than FK-violating or losing the clause.
3. **Never delete `defined_terms`** — they are deal-scoped; the term set survives the contract that happened to introduce a term. (`defined_terms.source_node_id` SET NULL if it pointed at a deleted node; the term itself stays.)

**Rejected:** **cascade-wipe everything the contract touched** (delete the deal-shared terms and cross-refs it created). Cleaner single-contract removal, but it silently breaks the *other* contracts in the deal — a deleted Offtake could take down a defined term or a cross-reference a sibling Licence still depends on. In a multi-contract deal that is data loss with no operator signal, which violates the "no silent corruption" bar the same way a bad node-match would.

**Rationale.** Contract deletion must not have deal-wide blast radius. Preserve-and-null keeps every surviving contract internally valid; a dangling cross-ref is a visible, recoverable marker, whereas a deleted shared term is irrecoverable. ## DD-64 — Mode B matching: LLM judges clause identity only; the redline delta is always deterministic

**Context.** The Phase-2 Mode B matcher (DD-28, spike plan in DEV_TODO) uses an LLM scorer on the ambiguous residue of clauses it can't match deterministically — to recognize a clause that is *the same clause, edited* (sentences added/removed, reworded, renumbered) versus a genuinely new/deleted clause. Lexical overlap drops when a clause is heavily edited, so a pure-threshold matcher mis-splits a heavily-edited-but-same clause into a delete + a new clause — the worst redline outcome (reads as "they replaced clause 7" when the truth is "they edited clause 7"). The LLM raises recall on that case. But the redline is staked on faithfulness to the exact counterparty wording (§2.1 content integrity, §2.4 credibility — the operator briefs a principal and screen-shares to a counterparty from it).

**Decision.** A hard division of labor between the LLM and the diff:
1. **The LLM scorer judges identity only** — its output is a **match decision (match / no-match / abstain) plus a confidence**, on the ambiguous residue only (the byte-identical and clearly-lexically-matched/unmatched clauses never reach it). It decides *whether two clauses are the same clause*; it does not decide, describe, or author *what changed*.
2. **The redline's change content is always a deterministic text diff** — once identity is fixed, the literal added/removed/reworded text is computed mechanically (exact delta), never LLM-authored or paraphrased. The LLM never writes a word that appears in the tracked-changes artifact.
3. **Abstain → operator confirms.** Confident match → auto-match (deterministic diff renders the edits); confidence in the abstain band → the clause is **flagged to the operator to confirm the match herself** (surfaced in structural triage, §11 step 6). The LLM buys recall on heavy edits; the abstain gate protects precision so the model can never silently merge two genuinely different clauses into one "edited" one.
4. A **plain-English "what changed" summary** (e.g. "they added two carve-outs to the indemnity cap") is permitted only as a **separate, clearly-labeled layer on top** of the redline — never a substitute for, and never mixed into, the exact tracked change.

**Rejected:** (a) **LLM authors or summarizes the change as the redline itself** — any model paraphrase silently drifts the redline from the counterparty's literal words; a redline's entire value is letter-faithfulness (§2.4). (b) **LLM scorer runs on every clause** — wasteful and slow; the deterministic funnel clears the bulk, the LLM is reserved for the residue (token-efficiency; cheaper model for a classification-shaped judgment). (c) **Auto-commit low-confidence matches to avoid bothering the operator** — a silent wrong match corrupts the diff invisibly (DD-35 high-consequence); abstaining to operator-confirm keeps Path-B matching in the low-consequence tier.

**Rationale.** AI is used where it beats string-matching (recognizing sameness through heavy editing) and nowhere else; the exact change is mechanical so the redline can never drift from the counterparty's words; uncertainty is surfaced to the operator rather than guessed. (The product owner confirmed the bound and the flag-uncertain-to-operator behavior.)

Blank/whitespace edits coerce to `NULL` (no language). A plain **[Use]** (no body) copies the stored draft verbatim, unchanged. **Rationale.** Editing-before-confirm is the strongest expression of the draft-vs-confirmed split: the export fields still receive only operator-approved language, and reusing the existing copy path (rather than a separate edited-write) keeps the draft row the one source of confirmed truth. **Rejected:** a separate PATCH-the-draft endpoint (adds a write path + a window where a stale edited draft sits unconfirmed) and writing edited text via `updateIssue` (it cannot reach the F31 export columns — only confirm may, per the addendum above).

## DD-69 — Donna advises/drafts on a grounded anchor, not on a surface (context-aware chat, F10b)

**Context.** DD-62 locked the Donna chat tab to read-and-explain only; F11 confined positional advice + counter-language to an open issue; F08d confined clause drafting to the ⋮ insert flow. Operating those three as separate surfaces means an operator drafting/editing a clause and wanting Donna's help hits the chat's deflection wall and must context-switch (create/open an issue) just to get advice — work shifted, not saved. **Decision.** Advice/drafting is gated on a **grounded anchor**, not on which surface the operator stands in. The reason chat deflected was never "chat is unsafe" — it was that an *ungrounded* positional assertion is the §2.4 risk; an issue was safe because it carries a concrete anchor (clause node + position ledger). So if the chat carries an equivalent anchor (a selected clause and/or an active issue), the same grounding exists and the same already-guard-railed engines (F11 advice, F08d drafting) apply. The chat becomes a **conversational front-end onto the existing safe engines**, not a new ungoverned advice surface.

**The boundary (three operations, three gates — the prompt rule):**
- **Drafting help** — allowed with a selected clause (revise it) or a description + deal context (new clause, F08d engine). Transient; operator commits via existing apply paths; bracketed placeholder for any unknown figure, never an invented number.
- **Positional advice** (recommend accept/reject/counter on the favorable→aggressive→deal-breaking spectrum, ask/settle/floor ladder) — allowed **only** with an active clause and/or issue in context (F11 engine), grounded in + cited to that anchor. No anchor → no advice.

Context unlocks commercial advice + drafting; it never unlocks a legal opinion.
- **No-context mode** — read-and-explain (F10); an advice request deflects **by acquiring context** ("tell me which clause — select it or open the issue"), not by refusing.
- **Always-on floor (every turn):** grounded & cited — **if she can't cite, she can't advise**; never fabricates a figure (recommends the structure, flags the missing benchmark, no live research in v1 — as F11); a clause in context with no stated position → **elicit before recommending** (DD-14 rule 3), never invent a stance.

**Context model.** Active context = `{0..1 issue} + {0..n clause nodes}`, shown as a sticky **context chip** in the chat (`Discussing: Issue #3 · clause 3.4.12 [×]`), set via the **Brainstorm hand-off** (issue + its clauses) or a manual **"Ask Donna about this clause"** action (Lilly: IN for v1), cleared via `[×]`. **Grounding rule:** context is a **pointer (node_id/issue_id) resolved LIVE from the DB each turn**, never a snapshot baked into the DD-40 window — so Donna always sees current clause text; the transcript is conversational memory, the grounding is always re-fetched.

**Brainstorm hand-off.** The F11 card's "Brainstorm ↗" navigates to the Donna tab, sets the chip to the issue + clause, and primes the opening turn with the issue's current `donna_recommendations` draft (fetched, not regenerated); the operator iterates ("more aggressive", "what's our floor?") and commits adopted language through the **same DD-68 confirm path** as the card's [Use Donna's language] — no second write path. Still linear per turn (retrieve → call → return); no LangGraph (DD-52 holds).

**Amends DD-62** (relaxes "read-and-explain only": the chat is read-and-explain *only when no context is set*; with a clause/issue anchor it may advise + draft via the F11/F08d engines). **Clarifies DD-14** (the four rules are unchanged — what changes is the advice *entry point*: "a grounded anchor exists", not "an issue exists"). **[Brainstorm-persistence half WITHDRAWN by DD-73, 2026-06-25:** Brainstorm does **not** persist and the ephemeral-overlay design is **not** superseded — Brainstorm stays ephemeral and Donna distills a stored summary on close. Only the context-aware-advice half of this DD stands.]**

**Rejected:** (a) advice anywhere in chat regardless of context — reintroduces the ungrounded-assertion risk DD-62 guarded; (b) overloading F10's locked read-and-explain row instead of a new F10b row — muddies a deliberately-locked scope; (c) a new from-scratch advice engine — the F11/F08d engines are already grounded + guard-railed + evaluated; chat reuses them via shared grounding, routed by context + intent.

**Rationale.** The guarantee that Donna never gives reckless ungrounded advice is preserved exactly — the advice rules and engines don't change, only the entry condition moves to the property that always made advice safe (a grounded anchor). The relaxed-boundary safety is proven by an eval (the gate): positional advice only with an anchor, legal opinions walled off even WITH context, no-context advice deflected-by-acquiring-context, drafts use placeholders not invented numbers. (The product owner directed the capability + signed off the two taste calls — manual clause→chat entry IN; the brainstorm-persistence call was **reversed to ephemeral-plus-stored-summary in DD-73**; product scoped the boundary; engineering owns the reuse/routing ADR.)

## DD-70 — Version presentation: monotonic v-numbers over snapshots; the working copy is never numbered or locked; an "edited since last send" marker, not a lock

**Context (operator, 2026-06-25).** The operator reasons in numbered versions ("I sent v3, they return v4") and intuitively framed sending as *lock the version → unlock to edit (with an "are you sure") → it becomes a draft again, carrying two tags (sent + working draft)*. That framing puts state on the wrong object: it would build a per-version lock/unlock state machine on top of the snapshot+pointer model that already exists (DD-22→DD-48, DD-61; F14 snapshot backend built), and — load-bearing — a "version you can unlock and edit" breaks snapshot immutability, which **is** the redline baseline (DD-48). If the sent snapshot can move, `diff(working_copy, last_shared_with_X)` no longer answers "what changed since they saw it" and the redline becomes a lie. Decision separates the operator's **cognitive unit** (a numbered version) from the **data unit** (immutable snapshots + named pointers + one live working copy).

**Decision.**
1. **The working copy is always live, never locked, and never numbered.** It is the contract as it currently is. A send does not lock it; editing after a send needs no "unlock." Numbers attach only to *frozen* artifacts, so the live tree — which changes on every keystroke — never carries a version number (a number that churns per edit means nothing).
2. **Snapshots are the only numbered artifacts, on one monotonic timeline.** Every **boundary event** mints the next integer: a **send** (→ to counterparty or legal) *or* a **receive** (← a revision imported from counterparty or legal). Each version is direction-tagged (`v3 → sent to counterparty Jun 12`, `v4 ← received from counterparty Jun 20`). This is the **version-lineage view** DD-48 anticipated (F27), now with operator-facing v-numbers. Between events the working copy reads as "vN + unsent edits" — it mints the next number only when the next boundary event freezes it.
3. **Sending freezes a faithful copy and stamps a pointer; subsequent edits never touch it.** On send, an immutable snapshot captures the tree as-is and `last_shared_with_X` → that snapshot; later edits land on the working copy + a `node_versions` row, and **the snapshot and pointer do not move** until the next send. The pointer is the redline baseline; intervening Internal/Copy-only grabs cut nothing (DD-61).
4. **"Edited since last send" is a passive marker, not a lock or a confirm.** When the working copy diverges from `last_shared_with_counterparty` (equivalently, unassigned `node_versions` exist since that pointer's snapshot), the contract shows a single consolidated, **non-blocking** "edited since last send" indicator beside the lifecycle badge. No unlock dialog, no per-edit "are you sure", no nudge — the operator edits freely mid-call; the marker just tells the truth that the recipient doesn't have the latest. (One consolidated marker in v1; per-party divergence detail lives in the lineage view, not as competing badges.) This is the correct form of the operator's "two tags": badge = last boundary event with the other side; marker = local divergence from it.
5. **Lifecycle-badge clarification (resolves an F27 ambiguity).** The F27 badge tracks the **last boundary event with the other side**, not local editing activity. Editing your own already-sent working copy does **not** flip `Sent to counterparty` → `Working copy` (renamed from `In flight` by DD-75); the badge stays `Sent to counterparty` (ball is still in their court) and the unsent-edits marker appears. `Working copy` is reserved for a fresh draft or an *engaged inbound revision* (something they sent back), never for post-send local edits.

> **Amended by DD-75 (2026-06-25 — wording only, no mechanism change).** The badge label **`In flight` is renamed `Working copy`** (the operator's term for a live, unsent draft) — the state's *meaning* is unchanged: it still covers a fresh draft and an engaged inbound revision, and post-send edits still keep the `Sent …` badge + raise the marker exactly as §5 specifies. DD-75 also makes §5 concrete as a deterministic FIRST-MATCH-WINS resolver (`services/lineage.derive_status`) and pins the marker reference to the **most-recent `last_shared_with_*` snapshot** (not `last_export_at`, which drives only the DD-72 mark-time warning) and the v-number to `ROW_NUMBER … ORDER BY created_at`. No DD-70/71/72 mechanism changes.

**No schema change.** Snapshots, the four named pointers, and `node_versions.snapshot_id` (null = unassigned edit) already exist (DD-48/DD-61, F14). The v-numbers are a presentation order over `contract_snapshots`; the marker is derived from unassigned `node_versions` since the `shared` pointer. Engineering owns the render + the divergence query (F27 lineage view + F21 version diff); no new entity.

**Rejected.** (a) **Literal per-version lock/unlock with dual sent+draft tags** — builds a state machine over data we already have and, fatally, lets a "sent" version be edited, breaking the immutable redline baseline (DD-48); the operator's outcome is delivered instead by always-frozen snapshots + an always-live working copy + the marker. (b) **Active nudge** ("you have unsent edits — re-send?") or **blocking confirm on edit** ("this was sent; edit anyway?") — friction exactly mid-call when the operator is fastest; re-sending is a deliberate act, not something to prompt. (c) **Numbering only outbound sends**, with counterparty returns as sub-items — conflicts with the operator's own "they return v4" mental model; one monotonic alternating timeline matches how she already counts. (d) **Numbering the working copy** — the number would change on every edit and stop denoting a real shared artifact.

**Rationale.** Reuses the snapshot+pointer model and the built snapshot backend untouched; preserves the immutability that makes redline and "what changed since they saw it" correct; and delivers the operator's exact mental model (numbered versions she sent/received, a clear "they don't have my latest" signal) as a presentation skin, not new machinery. (The product owner confirmed the model + signed off the two taste calls — passive marker over nudge/confirm, monotonic send-and-receive numbering — 2026-06-25; product framed the boundary; the presentation/divergence-query mechanism is engineering's ADR.)

## DD-72 — The "edited since last export" drift marker is a timestamp, not a tree-hash (engineering ADR for DD-71)

**Context (engineering, 2026-06-25).** DD-71 requires Mark-as-sent to warn when the working copy was edited since the last export ("you've edited since your last export — Mark anyway / Re-export"). DD-71 framed this as donna.ai "remembering the working-copy state at each export (a tree reference/hash) to compare against at mark-time," and asserted *no schema change* for the whole feature. Building it surfaced the real choice: what reference does the export persist, and is it worth a content-hashing subsystem for a **non-blocking** nudge?

**Decision.**
1. **One nullable column — `contracts.last_export_at TIMESTAMPTZ`** (migration `0005_contract_last_export_at.sql`, `ADD COLUMN IF NOT EXISTS`; folded into `schema.sql` + its `schema_migrations` seed per DD-57). The clean-copy export route stamps `last_export_at = now()` on every export.
2. **Drift = "any non-deleted node has `updated_at > last_export_at`" (or `last_export_at IS NULL`).** `nodes.updated_at` is already bumped on every edit/insert/move, so this is a cheap `EXISTS` probe at mark-time. Mark-as-sent returns `drift: bool`; the UI shows the non-blocking warning (mark-anyway / re-export) on `true`, marks silently on `false`.
3. **Timestamp-precision only — accepted imprecision.** A node edited then reverted to identical text still bumps `updated_at`, so it can **false-positive** the warning. This is acceptable: the warning is a non-blocking nudge, never a gate, and a spurious "you may have edits" is harmless (the operator clicks through). Drift is never silenced wrongly in the dangerous direction (a real edit is always caught).

**Rejected.** (a) **A content tree-hash / snapshot-fingerprint subsystem** persisted at each export and recompared at mark-time — eliminates the edit-then-revert false-positive, but is a whole hashing+storage mechanism (canonicalize the tree, hash, store, recompare) for a warning the operator can dismiss in one click; the cost/benefit doesn't clear the bar (§2.4 puts effort on correctness that *matters*, and this warning's correctness budget is "nudge, don't gate"). (b) **No schema change at all** (DD-71's stated hope) — there is genuinely no prior column that records "when did we last hand the operator the file"; `last_shared_with_X` records *marks*, not *exports*, and the two drift signals are deliberately different reference points (DD-71 "Relation to DD-70"). One timestamp column is the minimum honest mechanism.

**Note on DD-71's "no schema change."** DD-71 said the snapshot/pointer mechanism needs no schema change — still true (snapshots, the four pointers, and `node_versions.snapshot_id` all pre-exist; Mark-as-sent reuses them untouched). DD-72 adds the **one** column the *drift marker* specifically needs — orthogonal to the snapshot/pointer machinery.

**Rationale.** A nullable timestamp + an `EXISTS` over the already-maintained `nodes.updated_at` delivers the exact non-blocking signal DD-71 wants at near-zero cost, with one column and no new subsystem. The only imprecision (edit-then-revert) lands on the harmless side of a nudge that is never a gate. (Engineering ADR per DD-71's hand-off; the drift-comparison mechanism was explicitly delegated to engineering.)

## DD-75 — Lifecycle-badge derivation + lineage-view scope (F27 backend; implements DD-70 §5)

**Context (product + engineering, 2026-06-25).** DD-70 §5 fixed *what* the F27 lifecycle badge tracks (the last boundary event with the other side, not local editing) and DD-71/DD-72 built the boundary action (Mark-as-sent) + the drift timestamp. What remained was the **derivation**: given a contract's snapshots, the four DD-48 pointers, and its working copy, exactly which badge shows, with which version number, and when the "edited since sent" marker is set — plus the scope of the lineage *view* (the v1→…→vN render) for v1. This is the F27 backend (badge resolver + lineage API); the frontend, the Phase-2 received-pointer data, and the F21 version diff remain pending.

**Decision.**
1. **Operator taxonomy (verbatim).** `Working copy` · `Sent to counterparty` · `Sent to legal` · `Sent to counterparty & legal` · `Your move` · `Signed`, plus a single passive **"edited since sent"** marker. This **renames DD-70 §5's `In flight` → `Working copy`** — wording only, the state's meaning is unchanged (a fresh/unsent draft or an engaged inbound revision).
2. **Badge derivation — top-down, FIRST MATCH WINS** (LBE = latest boundary event = most-recent snapshot, tagged by the pointer(s) resting on it):
   1. `contracts.status = signed` → **"Signed · vN"**.
   2. No snapshot exists → **"Working copy"**, no version number.
   3. LBE is a **receive** (Phase-2) and not engaged (no edits since the receive snapshot) → **"Your move · vN"**.
   4. LBE is a **receive** and engaged (edited since) → **"Working copy"** (based-on "vN received from X").
   5. LBE is a **send** → **"Sent to counterparty · vN" / "Sent to legal · vN" / "Sent to counterparty & legal · vN"** (one snapshot can hold both `shared` pointers).
   **Post-send-edit edge:** if LBE is a send and the working copy has diverged since that snapshot → the badge **stays "Sent to … · vN"** and the **"edited since sent" marker is set** ("Sent to counterparty · v3 · edited since sent"). It never reverts to Working copy (DD-70 §5). In v1 the LBE is always a send (only Mark-as-sent cuts snapshots, DD-71), so rules 3–4 stay dormant until Mode B adds receives.
3. **Marker reference.** The marker (and the rule-3/4 "engaged" test) compares the working copy to the **most-recent `last_shared_with_*` snapshot** (counterparty or legal, whichever is newer): drift = any non-deleted node with `updated_at >` that snapshot's `created_at`, or an unassigned `node_versions` row since it. It is **NOT** `contracts.last_export_at` — that timestamp drives only the transient DD-72 mark-time drift warning. One consolidated boolean marker, not per-party badges.
4. **Version numbering.** vN = the snapshot's position via `ROW_NUMBER() OVER (PARTITION BY contract_id ORDER BY created_at)` over **ALL** snapshots (sends now, receives later) — so numbers never renumber when Mode B interleaves receives. The latest snapshot's number = the snapshot count. **The working copy is never numbered.**
5. **Lineage-view v1 scope.** The view renders the numbered send/receive timeline (working copy marked separately, never numbered), supports **read-only open** of any snapshot (a render adapter rebuilds the cockpit tree shape from the frozen JSONB dump, soft-deleted nodes dropped), and shows the **two `received` pointer slots greyed** (not populatable until the Phase-2 revision-import path). **Out of v1 scope:** the F21 version-to-version *diff* and any *restore*/revert.

**No schema change.** Snapshots, the four DD-48 pointers, `node_versions.snapshot_id`, `nodes.updated_at`, and `contracts.status` all pre-exist. The badge, the timeline numbering, and the marker are pure derivations.

**Implementation.** `services/lineage.py`: `derive_status` (pure, I/O-free resolver — the `diverged` signal is supplied by the caller so it stays DB-free), `derive_status_for_contracts` (set-based badge for the My-Contracts + home list — ONE query: LBE + its pointer tags + a per-contract divergence-EXISTS, no N+1), and `get_lineage` (badge + numbered timeline + working-copy marker + reserved slots). `services/snapshot.py` adds the read-only render adapter (`get_snapshot_tree`) + list helpers (`list_numbered_snapshots`, `list_pointers`). `api/lineage.py`: `GET /contracts/{id}/lineage` and `GET /contracts/{id}/snapshots/{snapshot_id}/tree`. The set-based badge is wired into the contract-list read so every card carries its `badge`.

**Relation to prior ADRs.** **Implements DD-70 §5** as a deterministic resolver and **amends DD-70** with the `In flight`→`Working copy` rename (wording only). **Changes no mechanism in DD-70/DD-71/DD-72** — Mark-as-sent, the pointers, and the `last_export_at` drift warning are untouched; this ADR only *reads* them. **Builds on DD-48** (the four pointers + the anticipated lineage view).

**Rejected.** (a) **`last_export_at` as the marker reference** — it records exports (grabs), not sends; the marker must compare to the last *marked-sent* snapshot, or a mid-draft re-export would clear a real "they don't have my latest" signal. (b) **Per-version lock state / numbering the working copy** — already rejected in DD-70; restated here because the resolver makes it concrete (the working copy carries no version). (c) **N+1 per-card badge query** — the list path resolves all badges in one set-based query. (d) **Including the F21 diff or restore in the lineage view** — diff is its own feature (F21); restore is out of v1 (snapshot immutability is the redline baseline).

**Numbering note.** This ADR is **DD-75**, not DD-74 as the build brief labeled it — DD-74 ("v1 is local-only, permanently") was already committed, so the next free number was taken to avoid clobbering it. (Engineering ADR per DD-70's hand-off — product scoped the taxonomy + the lineage-v1 cuts; engineering owns the resolver, the marker reference, the numbering query, and the set-based read.)

## DD-78 — Mode-B abstain match-confirm lives in a structural-foundation phase; F03c content review is one document-ordered stream, not four bucket-screens (F03c)

**Context (product, 2026-06-25).** The Mode-B revision matcher (DD-64, built + de-risked) sorts a clean counterparty revision into four buckets — **matched-with-edits**, **new**, **deleted**, and **abstain** (pairs below the confidence/margin gate it deliberately punts on rather than risk a wrong auto-match). F03c is the next build. Three product calls were owed at the engineering boundary: (1) where the abstain match-confirm step lives; (2) the acceptable confirmations-per-import ceiling before Donna stops beating Word; (3) how the four buckets are sequenced and presented.

**Decision 1 — match-confirm is a sub-step of the structural-foundation phase (§11 step 6b), not folded into depth/heading triage and not a free-standing top-level stage.** Confirming an abstained pair and fixing a node's depth are both *"confirm Donna's structural read before content"* (DD-30 — structure precedes content, deliberately separated), so they share a phase; but their **action vocabularies differ** (± level / heading-toggle vs. confirm-match / it's-new / match-different), so they are **separate sub-screens** (6a tree-shape, 6b match-confirm), never one undifferentiated widget — the same anti-conflation reasoning DD-30 applies between structure and content. Abstains **must** resolve up front, not inline in the content stream: an abstained pair has no diff to render until confirmed, and resolving one abstain can reclassify another node (a rejected match drops its guessed counterpart to *deleted*), which would retroactively churn a content stream already in progress. **Operator-facing 6b:** a ranked queue (most-uncertain first) of cards — incoming clause beside Donna's best-guess baseline counterpart + confidence + one-line reasoning — with **[Confirm match]** / **[Not a match → it's new]** / **[Match to a different clause]**. Each resolution reclassifies the pair into matched / new / deleted.

**Decision 2 — load ceiling: no blocking cap; two deliberately-separate thresholds + an escape hatch** (corrects the earlier PM note that "the matcher's gate is the one number"). The abstention *rate* and the operator's absolute *confirmation load* are different quantities and must not be conflated: a counterparty who rewrites the whole contract throws a huge confirmation load no matter how good the matcher is. So: (a) **matcher abstention gate (≤~15% of changed nodes) is engineering's quality metric** — exceed it and the matcher is regressing, fixed headless against the gold map, not a UX problem; (b) **the ranked + bulk-actionable queue is the product effort-guard** — "confirm all remaining high-confidence" / "treat rest as new" (the DD-34 two-tier-triage pattern, already blessed for the analogous 700-tracked-change Mode-A case) so load scales sub-linearly with diff size; (c) **rewrite-scale escape hatch** — when matched+new+abstain span the majority of the tree (or abstains alone blow past the gate), Donna surfaces a non-blocking "re-import as a new baseline (Mode A)?" banner. The honest answer to "has Donna lost its value past N confirmations": at rewrite scale, *diffing is the wrong tool* — so Donna offers the right one rather than making the operator grind through hundreds of matches. The floor comparison is not "redline in Word" but "Word Compare" (which on a clean copy produces the same ambiguous auto-matches, just buried in a garbled diff instead of surfaced) — so a bounded, ranked, explicit abstain-confirm is strictly better than the Word baseline, not overhead on top of it.

**Decision 3 — content review is ONE document-ordered linear stream with type-appropriate cards, not four bucket-screens.** Once 6b settles every abstain, the three reviewable types (matched-with-edits / new / deleted) flow top-to-bottom in document order — the operator's cognitive unit is *"walk the contract,"* not *"batch all deletions"*; document order preserves the narrative of what the counterparty did (a deletion at clause 4 next to an edit at clause 5 are related moves). Each item renders a card on the DD-26 inline-markup + DD-27 four-action pattern, adapted per type (edit: Accept theirs / Use Donna's / Edit / Keep · added: Accept addition / Use Donna's / Edit / Reject · deleted: Accept deletion / Reject / Edit). Donna pre-classifies significance (`trivial`|`substantive`) for added and deleted nodes as well as edits, so trivial ones are pre-accepted. A new node already carries its incoming-tree position, so there is no separate "placement" action — the decision is purely accept/reject the counterparty's addition.

**Gap closed.** Pre-DD-78 §11 described the review UI (step 8) only for *hunks* (edits); **new and deleted nodes were identified (step 3) but had no specified review path.** DD-78 routes them through the same change-row + decision-unit staging (each as a single whole-node decision) and the same content stream.

**Rejected.** (a) **Fold match-confirm into the step-6a depth/heading widget** — different action vocabulary; mixing "± level" with "confirm this is the same clause" recreates exactly the structure-vs-content confusion DD-30 forbids. (b) **A free-standing top-level match stage outside the structural phase** — over-structures a step that shares 6a's purpose and mental model; it belongs *in* the structure-before-content phase. (c) **Resolve abstains inline as the operator reaches them in the content stream** — impossible (no diff to show until matched) and unstable (one resolution reclassifies others mid-flight). (d) **A single magic confirmations-count cap that blocks the import** — conflates the matcher quality metric with the product value threshold and punishes legitimately-large diffs; replaced by the two-threshold + escape-hatch model. (e) **Four separate bucket-screens (all news, then all deletes, then all edits)** — breaks document-order narrative and forces the operator to re-orient four times; rejected for the single ordered stream.

**Amends §11 Mode B** (step 3 Path B four-bucket vocabulary; step 4 escape hatch; step 5 new/deleted staging; step 6 → 6a/6b; steps 8–9 typed cards + decisions). **Builds on DD-64** (matcher buckets + abstain), **DD-26** (inline rendering), **DD-27** (four actions), **DD-30** (structure before content), **DD-34** (bulk-triage pattern for load). (Product owns the phase placement, the sequencing, and the load model; engineering owns the matcher abstention gate value, the schema extension for whole-node add/delete decisions on the change-row model, and the ranked-queue implementation.)

## DD-79 — F03c `change_kind` is DERIVED from F03b's staged columns; no migration for the four-bucket review vocabulary (F03c)

**Context (engineering, 2026-06-25).** DD-78 asks F03c to present four change kinds — **edited** (matched with hunks), **new**, **deleted**, **abstain** — each with its own decision vocabulary. ## DD-85 — Deleting a contract version wipes it completely; deleting the latest rolls the working copy back (cockpit version management, 2026-06-26)

**Context.** The operator needs to delete/wipe past versions from the cockpit lineage — to clear test/mistake versions and to undo work back to an earlier sent version. Requested 2026-06-26; operator chose "Behavior B" (rollback) over cleanup-only.

**Decision.** A version (an immutable lineage snapshot) can be **hard-deleted ("wiped completely")**. Delete behaves by position:
- **Deleting the LATEST version** (the one the working copy currently sits on) **rolls the working copy's content back to the now-latest remaining version.** Example: on v4 (Clause 5 = "9%"), delete v4 → working copy reverts to v3's content (Clause 5 = "12%"); v4 is wiped; next Mark-as-sent re-mints v4.
- **Deleting a NON-LATEST (middle) version** removes only that version from the lineage — **working copy untouched** — leaving a **preserved gap** (delete v2 → lineage v1, v3, v4; next Mark-as-sent = v5). Reverting to a middle version's predecessor would destroy unrelated current work, so middle-delete never rolls back. *(The latest-vs-middle split is derived from the operator's two examples — "delete v4 → v3 becomes working copy" and "delete v2 → gap v1,v3,v4"; confirm at build.)*
- **Deleting the only/last remaining version** → snapshot removed, contract returns to the never-sent "Working copy" state; nothing to roll back to, working copy untouched.

**Guardrails (non-blocking warnings, operator confirms — mirrors DD-71/DD-72 drift pattern):**
- **Rollback is destructive** — deleting the latest discards the working copy's current content + any unsent edits since the prior version. Warn first.
- **Deleting an already-sent version** (carries a `last_shared_with_*` pointer) **erases the record of what was sent** to that party and rolls the **redline baseline** back to the previous shared snapshot. Warn (name party + date).

**Schema/engineering implication (amends DD-70/DD-75 — Kevin's ADR).** Gap-preservation is **incompatible with DD-75's `ROW_NUMBER()`-derived v-numbers** (ROW_NUMBER would renumber v3,v4 → v2,v3 after deleting v2, killing the gap). Version numbers must be **persisted** (stored `version_number` on `contract_snapshots`, monotonic from max+1, never reused — latest-delete re-mint is just max+1 of the remainder). This changes the DD-70/DD-75 numbering basis from derived to stored (schema change). Delete cascade must: clean the snapshot's `node_versions.snapshot_id` group, roll back any `last_shared_with_*` / redline-baseline pointer to the previous snapshot, and (Phase 2) handle any Mode-B `received` pointer referencing the deleted snapshot; rollback = reconstruct the working tree from the now-latest snapshot's content (extends the DD-63 delete cascade discipline). Engineering owns the mechanism.

**Rejected.** Behavior "A" (cleanup-only, never touch the working text) — operator explicitly wants latest-delete to revert her document. Soft-delete/keep-history — operator wants a complete wipe.

**Builds on / amends DD-48 / DD-70 / DD-75; new schema (persisted `version_number`).**

## DD-87 — Version-delete mechanism: persisted `version_number`, UPDATE-restore rollback, FK-correct snapshot cascade (engineering ADR for DD-85)

**Context (engineering, 2026-06-26).** DD-85 (operator's "Behavior B") authorized hard-deleting a lineage version and rolling the working copy back on a latest-delete, and named the schema implication (persisted v-numbers, gap-preservation). This ADR pins the *mechanism* DD-85 delegated to engineering: the migration, the numbering swap, the rollback/tree-restore algorithm, the FK-correct cascade, and the API contract. Owns no product behavior — that is DD-85.

**Decision.**

1. **Persisted `version_number` (numbering basis derived→stored — amends DD-70/DD-75).** New `contract_snapshots.version_number INTEGER NOT NULL` + `UNIQUE (contract_id, version_number)` (migration `0012_snapshot_version_number.sql`, `ADD COLUMN IF NOT EXISTS` + backfill `ROW_NUMBER() OVER (PARTITION BY contract_id ORDER BY created_at)` over existing rows + `SET NOT NULL` + the unique constraint; folded into `schema.sql`'s `contract_snapshots` CREATE and its `schema_migrations` seed). Backfill reproduces the exact old `ROW_NUMBER` values, so existing lineages are visually unchanged. New numbers mint as `COALESCE(MAX(version_number),0)+1` per contract, computed atomically inside the INSERT (`INSERT … SELECT … FROM contract_snapshots WHERE contract_id=$1`). **Both** snapshot-insert paths assign it (`cut_snapshot` = Mark-as-sent; `snapshot_tree` = Mode-B as_received). Readers swapped off `ROW_NUMBER` to the column: `snapshot.list_numbered_snapshots`, `lineage._compose_badge`/`derive_status`/`derive_status_for_contracts` (badge v-number = `MAX(version_number)` of remaining, not the snapshot count — they diverge after a delete), `mark_sent` preview (`MAX+1`, not count+1). **Why stored:** `ROW_NUMBER` renumbers survivors after a middle-delete (v3,v4 → v2,v3), destroying DD-85's required gap; a persisted, never-reused integer preserves it. *Rollback path for this one-way door:* drop the column + UNIQUE, revert readers to `ROW_NUMBER` — valid only while no delete has created a gap; once a gap exists the derived basis can't represent it (which is the whole point).

2. **Delete is position-typed off `version_number` vs `MAX(version_number)`.** `latest` = the max-version snapshot (in v1 the working-copy basis). `middle` = any non-max. `only` = count == 1.

3. **Latest-delete rollback = UPDATE-restore, never re-insert.** Soft-delete is the node model (`node_delete` flips `is_deleted`, never removes the row), so a snapshot's node ids are all still live rows. Restore to the now-latest remaining snapshot (`get_snapshot`'s frozen JSONB): for each snapshot node, `UPDATE nodes SET parent_id, order_index, content_type, heading, body, is_deleted, updated_at = <snapshot.created_at>`; any live node absent from the snapshot (added after it) → `is_deleted = true`. Then **DELETE pending `node_versions` (`snapshot_id IS NULL`)** for the contract (= discard the unsent edits DD-85 says a rollback discards). Stamping `updated_at = snapshot.created_at` (not `now()`) keeps the DD-75 divergence probe from false-flagging "edited since sent" on freshly-restored content (a rollback is the DD-72 edit-then-revert case; this just makes it clean). **Oracle (free inverse-pair):** after restore, `get_snapshot_tree(now-latest)` must equal the live `get_contract_tree` — asserted in a system test.

4. **FK-correct cascade to `contract_snapshots`** (extends DD-63 discipline; every FK into the table handled before the row drops): (a) `node_versions.snapshot_id == deleted` → DELETE the group (DD-85); (b) `snapshot_pointers.snapshot_id == deleted` (NOT NULL) → roll back to the deleted snapshot's predecessor (next-lower `version_number`), or DELETE the pointer row if none (only/last); (c) `issues.opened_in_snapshot_id`/`resolved_in_snapshot_id` (nullable) → SET NULL; (d) `counterparty_revision_sessions.baseline_snapshot_id` (NOT NULL, Phase-2) → **defensive 409 guard**: refuse the delete if a session's baseline is the target (can't null a NOT-NULL baseline; not reachable in v1 data). Middle/only delete = the same cascade minus the working-copy restore. Pointer rollback also covers the DD-48 `received` pointer defensively though no data populates it yet.

5. **API — `DELETE /contracts/{id}/snapshots/{snapshot_id}?confirm=<bool>` (mirrors Mark-as-sent's acknowledge two-call).** `confirm=false` (default) → a **preview** (no mutation): `{deleted:false, is_latest, will_rollback, rollback_to_version, sent_record:{party,date}|null, warnings:[…], …}`. `confirm=true` → executes, returns `{deleted:true, rolled_back, pointers_rolled_back, …}`. `sent_record` is set when the target carries a `shared` pointer (names party + snapshot date for the DD-85 warning). 404 if the snapshot isn't the contract's; 409 on the baseline guard. Audit `version_deleted`.

**Rejected.** (a) Re-insert/replace the node set on rollback — needless churn of the many `nodes` FK children (node_versions/footnotes/cross_references/parameter_references/embeddings); UPDATE-restore is FK-free given soft-delete. (b) `updated_at = now()` on restore — would false-flag the "edited since sent" marker on content identical to the send. (c) Keep `ROW_NUMBER` numbering — incompatible with gap preservation (see §1). (d) DELETE-with-body for the confirm flag — query param is simpler and matches the existing `deleteJson` client helper.

**Builds on / amends DD-70 / DD-75 (numbering basis) and DD-63 (cascade); implements DD-85.**

## DD-89 — Cross-document recommendation consistency: cluster identical counterparty changes and judge once (F34, 2026-06-28)

**Context.** Donna's counterparty-revision recommendations are generated per-hunk: `recommend_session` loops every change's hunks and calls `_analyze_hunk` once each — an independent LLM request at temperature 1.0 (Opus rejects 0.0), no shared state between calls, grounded only on the single clause's subtree. Consequence: the SAME counterparty edit appearing in multiple clauses (e.g. a rate change repeated across several clauses, or a defined-term rename appearing throughout the document) is judged N independent times and can receive N different verdicts and counter-language — Donna can contradict herself within one contract. No clustering, no cross-hunk memory, no propagation exists.

**Decision.** v1 implements **"A" — cluster-and-judge-once.** Before recommending, identical/related counterparty changes are grouped into a cluster (same figure, same defined-term rename, or same original→proposed edit); Donna judges each cluster ONCE and the single verdict + counter-language applies to every occurrence — making consistency a structural property, independent of sampling temperature. In the review surface a cluster collapses into a **single grouped stop** ("this change appears in N clauses"): the operator decides once (accept/reject/counter), the decision applies to all members, with an **expand to peel off and override an individual clause**.

**Rejected.** (a) **Lower the temperature for determinism** — Opus 4.8 rejects 0.0, and even pinned sampling wouldn't make two *separate* calls agree; clustering fixes the cause, not the symptom. (b) **Per-clause stops with an "apply to all?" confirm** — more clicks, and the operator re-reads the same change N times; the grouped stop is the real effort-saver (Lilly's pick). (c) **"B" coherence sweep in v1** (flag untouched stale occurrences of a changed term/figure elsewhere) — higher value but Donna would propose edits *beyond* the counterparty's redline over a whole-document read, a materially larger hallucination surface; **named and deferred as a fast-follow, not dropped.**

**Engineering owns (boundary calls for Kevin's ADR):** the cluster-detection mechanism (how "the same change" is normalized — exact original→proposed match vs figure/term equivalence vs semantic), where judge-once runs (a clustering pre-pass before the per-hunk loop, writing one recommendation fanned out to all members), and how a grouped stop renders + propagates a decision to all members with a per-clause peel-off override.

**Builds on** DD-64 (deterministic redline delta, LLM judges identity) **and DD-83** (per-change recommendation engine). F34.

**Grouped-stop after peel-off (2026-06-28, Lilly).** When the operator expands a grouped stop and overrides one member differently, the cluster renders as a **mixed summary** (e.g. "4 rejected · 1 accepted") naming the overridden clause — decided-with-exception, NOT reverted-to-open (preserves the decide-once benefit while making the divergence visible at the group level).

## DD-94 — Deleting a version discards its in-progress review: cascade + preview warning (2026-06-28)
